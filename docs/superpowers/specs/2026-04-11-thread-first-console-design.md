# Thread-First Console 改版设计文档

## 1. 背景

当前 `console/` 已经具备以下核心能力：

- 线程列表与新建线程
- 线程工作区
- 机器状态查看
- 环境资源管理：`skill / mcp / plugin`
- Agent 配置管理：全局默认配置、机器覆盖配置、配置下发
- 基于 WebSocket 的实时事件刷新
- 审批流转
- Turn 启动、`steer`、`interrupt`

但当前 Console 的信息架构仍偏向“模块式控制台”：

- `Overview`
- `Machines`
- `Threads`
- `Environment`
- `Settings`
- `Thread Workspace`

这种结构能覆盖功能，但不够适合高频线程操作者的主工作流。新版 Console 需要在保留已实现能力的前提下，重构主次关系和交互组织方式。

## 2. 设计目标

新版 Console 的目标不是做一个传统运维后台，而是做一个 `Thread-first` 的统一工作台。

最重要的用户目标是：

1. 快速回到自己正在处理的线程
2. 在单个工作区内持续观察流式输出
3. 在当前上下文中直接处理审批
4. 在当前上下文中执行 `steer / interrupt`
5. 在多个线程之间快速切换并继续工作

对应的核心设计要求：

- Thread 成为产品主对象，而不是某个列表页中的条目
- Workspace 成为产品主舞台，而不是二级详情页
- 审批成为消息流的一部分，而不是独立侧栏任务
- 机器、环境、配置降级为支持层入口

## 3. 目标用户

新版 Console 的主要用户是：

- 高频使用线程进行协作、观察和控制的研发或操作者

需要兼顾但不是主要优化对象的用户是：

- 负责机器、环境、配置稳定性的运维或平台角色

设计优先级必须服从主要用户的高频工作流，不能回到“先看总览，再进线程”的后台式结构。

## 4. 已确认的产品方向

经过讨论，当前版本已明确以下决策：

- 新版 Console 以 `Thread / Workspace` 为中心
- 以已实现功能为主，但允许为下一阶段能力做少量前瞻预留
- 主要服务对象是高频线程操作者
- 最优先优化的主路径是：
  - 进入线程后高效处理流式输出、审批、`steer`、`interrupt`
- 工作区气质采用 `Hybrid`：
  - 仍以消息流为中心
  - 但将关键控制能力强融合进工作流中
- 顶层结构采用 `Thread Hub`
- 取消右侧常驻栏
- 审批、Turn 状态、`steer / interrupt` 融入中间消息流
- 左侧线程栏可折叠，默认展开
- 左侧线程栏按机器分组
- `Machines / Environment / Settings` 放在左侧线程栏底部作为低频入口
- 打开 Console 时默认回到上次工作的线程

以上内容作为设计阶段的约束条件，不再重新讨论为“Dashboard-first”或传统模块式首页。

## 5. 新版信息架构

### 5.1 一级结构

新版一级结构建议如下：

1. 默认进入 `Thread Hub`
2. `Thread Hub` 承载线程目录与线程工作区
3. `Machines / Environment / Settings` 作为低频管理入口
4. 弱化或取消传统 `Overview` 首页的主入口地位

### 5.2 默认入口规则

默认入口规则如下：

- 打开 Console 时，优先回到“上次工作的线程”
- 如果最近线程不可用，则退化到 `Thread Hub` 选择态
- 如果当前没有线程，则进入线程空状态
- 如果当前没有机器接入，则进入无机器空状态

设计上必须把“继续工作”作为默认心智，而不是“先查看系统总览”。

## 6. 主布局要求

新版主布局采用双区结构：

1. 左侧线程栏
2. 中间单中枢工作区

不再保留右侧常驻审批/上下文栏。

### 6.1 左侧线程栏

左侧线程栏是线程导航区，不是运维侧栏。要求如下：

- 默认展开，支持折叠
- 按机器分组展示线程
- 每个机器分组需要表达：
  - 机器名称
  - 在线状态
  - runtime 状态
- 每个线程条目需要表达：
  - 标题
  - 线程状态
  - 当前选中态
- 顶部提供新建线程入口
- 支持未来扩展：
  - 最近访问
  - 收藏线程
  - 搜索/筛选
- 底部固定低频入口：
  - `Machines`
  - `Environment`
  - `Settings`

设计重点不是把线程栏做得像“资源树”，而是让用户快速定位和切换工作上下文。

### 6.2 中间工作区

中间区域是唯一主舞台，所有强相关操作尽量沿同一阅读轴展开。

建议结构：

1. Thread Header
2. 消息流 / 事件流
3. 底部输入区

Thread Header 只保留关键摘要信息：

- 线程标题
- `threadId`
- 所属机器
- 机器在线状态
- runtime 状态
- 当前是否存在活跃 Turn

更深层的机器、环境、配置内容不在工作区常驻展示。

## 7. Thread Workspace 设计要求

Thread Workspace 是本次改版的核心页面。

### 7.1 单消息流原则

以下内容必须统一承载在中间消息流中：

- 用户输入
- Agent 流式输出
- Turn started / completed / failed / interrupted
- 审批请求
- 审批结果
- `steer` 入口
- `interrupt` 入口
- 关键系统提示

设计原则：

- 审批不是跳出式任务，而是当前执行过程的一部分
- Turn 控制不是独立模块，而是执行流中的即时操作
- 系统事件不是单纯日志，而是工作流中的状态节点
- 用户视线尽量不离开中间区域

### 7.2 审批表现

审批请求应以内联卡片形式插入消息流，要求：

- 明确表达审批原因或命令摘要
- 如果存在问题项，直接在卡片内完成输入或选择
- 卡片内提供：
  - 接受
  - 拒绝
  - 取消
- 审批结果也应在消息流中留下状态反馈，而不是悄然消失

### 7.3 Turn 控制表现

当存在活跃 Turn 时，工作区需要在消息流中出现可操作节点或操作条，支持：

- 输入 `steer`
- 发送 `steer`
- 中断当前 Turn

设计上应保证：

- 操作离当前执行上下文足够近
- 不需要用户把视线切到另一个独立区域
- 活跃 Turn 与普通历史消息能被明显区分

### 7.4 输入区

底部输入区负责发起新一轮输入，要求：

- 常驻底部
- 明确可见
- 不与 `steer` 混淆
- 在无活跃 Turn 和有活跃 Turn 两种状态下都清晰表达当前行为

## 8. 低频管理模块定位

### 8.1 Machines

`Machines` 页面继续保留，用于查看：

- 机器在线状态
- runtime 状态

它属于运行状态管理页，不再承担主首页角色。

### 8.2 Environment

`Environment` 页面继续保留，用于管理：

- `skill`
- `mcp`
- `plugin`

当前已实现的启用、停用、安装、卸载、删除、编辑、详情查看等动作继续保留。

设计角色是“支撑线程运行的环境管理页”，而不是主舞台。

### 8.3 Settings

`Settings` 页面继续保留，用于管理：

- Agent 全局默认配置
- 机器覆盖配置
- 配置下发

它属于平台配置页，设计风格可以偏管理型，但不应干扰线程主工作流。

## 9. 空状态与异常状态

设计必须覆盖以下场景：

- 首次进入，没有最近线程
- 最近线程已不存在
- 最近线程所属机器离线
- 当前没有任何线程
- 当前没有任何机器接入
- 当前线程存在异常状态
- 当前 Turn 执行失败或中断

空状态或异常状态不能只给一行说明，必须配套明确动作，例如：

- 新建线程
- 返回线程 Hub
- 查看机器状态
- 等待机器接入

## 10. 少量前瞻预留

本次允许做少量下一阶段能力预留，但不能影响当前已实现能力的落地。

可预留方向：

- 最近访问线程
- 收藏线程
- 更强的线程搜索和筛选
- 更丰富的线程事件展示
- 更细粒度的审批上下文信息
- 更多线程排序方式

本轮不建议强化的方向：

- 复杂全局 Dashboard
- 重运营式首页
- 与当前实现差距过大的 IDE 化重设计

## 11. 给设计人员的交付范围

设计至少需要覆盖以下页面或状态：

1. `Thread Hub - 最近线程直达态`
2. `Thread Hub - 无线程空状态`
3. `Thread Hub - 左侧线程栏展开态`
4. `Thread Hub - 左侧线程栏折叠态`
5. `Thread Workspace - 普通流式输出态`
6. `Thread Workspace - 流内审批态`
7. `Thread Workspace - 活跃 Turn 的 steer / interrupt 态`
8. `Thread Workspace - 异常 / 中断 / 完成态`
9. `Machines`
10. `Environment`
11. `Settings`

同时需要补充交互说明：

- 左侧线程栏展开/折叠规则
- 线程切换后的反馈方式
- 不同消息类型的视觉层级
- 审批卡片在消息流中的插入方式
- 活跃 Turn 控件的视觉位置和激活条件
- 机器离线、runtime 异常、thread 异常的表达方式

## 12. 设计评审标准

设计方案是否达标，重点看以下问题：

1. 用户是否可以默认直接回到工作线程，而不是先进入后台首页
2. 用户是否能在一个连续界面中完成输出观察、审批、`steer`、`interrupt`
3. 审批是否真正融入消息流，而不是以另一种形式重建右侧栏
4. 左侧线程导航是否足够适合高频切换线程
5. `Machines / Environment / Settings` 是否被正确降级为低频支持层
6. 整个产品是否更像工作台，而不是传统管理后台

如果上述问题有明显偏差，则说明设计仍然没有真正完成 `Thread-first` 改版。
